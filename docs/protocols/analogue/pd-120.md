# PD120 protocol evidence

Status: **accepted for offline diagnostic-pattern and prepared-image transmission only**

This record freezes the evidence used by the M1E PD120 offline encoder. It does not
establish receive interoperability, live audio, live transmission, FSK ID, or support for
another PD-family mode.

The original bulletin defines PD120 as 640 by 496, VIS 95, and 248 transmitted line
pairs. Each pair is a 20 ms sync, a 2.08 ms porch, then equal-duration Y0, R-Y, B-Y, and
Y1 scans without intervening pulses or porches. Y0 and Y1 preserve source lines zero and
one respectively, while both colour-difference scans are the vertical average of those
two source lines. The pattern repeats for each consecutive pair. The table gives 190 us
per pixel, 508480 us per pair, and 126.103040 seconds for the image.

source pixel first, truncates the converted code toward zero, then vertically averages
each pair of converted colour-difference codes with integer division. It retains one

blue difference, Y1.

## Resolved wire values

| Field | Exact value | Evidence |
| --- | --- | --- |
| Stable ID and name | `pd-120`, PD120 | Original bulletin; project naming policy |
| VIS code | 95 (`0x5f`), seven bits | Original bulletin; Dayton; three encoders |
| VIS order and parity | Least-significant bit first; even parity | Dayton; shared accepted VIS evidence |
| Dimensions | 640 by 496 sRGB RGB8 input pixels | Original bulletin; Dayton; three encoders |
| Pair geometry | 248 groups, source rows `2p` and `2p+1` | Original bulletin; SSTV Encoder |
| Colour definition | Limited-range BT.601-derived luma Y, red difference, blue difference from nonlinear RGB | Dayton Appendix B |
| Luma conversion | `trunc(16 + (65.738R + 129.057G + 25.064B) * 0.003906)` | Dayton; SSTV Encoder |
| Red-difference conversion | `trunc(128 + (112.439R - 94.154G - 18.285B) * 0.003906)` | Dayton; SSTV Encoder V |
| Blue-difference conversion | `trunc(128 + (-37.945R - 74.494G + 112.439B) * 0.003906)` | Dayton; SSTV Encoder U |
| Conversion clamp | Clamp the real result to 0 through 255, then truncate toward zero | SSTV Encoder |
| Chroma sampling | Convert first; average rows `2p` and `2p+1` independently at every x; integer division by two | Original bulletin; SSTV Encoder |
| Pair schedule | 1200 Hz sync, 1500 Hz porch, Y0, red difference, blue difference, Y1 | Original bulletin; Dayton; three encoders |
| Sync | 20 ms at 1200 Hz | Original bulletin; Dayton; SSTV Encoder; PySSTV |
| Porch | 2.08 ms at 1500 Hz | Original bulletin; SSTV Encoder; PySSTV |
| Component scans | Four scans of 121.6 ms, 640 values each, no separators | Original bulletin; Dayton; SSTV Encoder |
| Component mapping | Linear: `1500 + 800 * code / 255` Hz | Dayton Appendix B |
| Pair duration | 508.48 ms | Exact selected interval sum |
| Image duration | 248 times 508.48 ms = 126.103040 s | Original bulletin |
| Full duration | 910 ms VIS header plus image = 127.013040 s | Exact calculation |

PD120 data bits for code 95 are `1, 1, 1, 1, 1, 0, 1` on the wire. They contain six
ones, so the even-parity bit is zero.

## Deterministic arithmetic

The published decimal coefficients are represented as exact signed integers over one
billion after multiplying by the exact decimal factor 0.003906. Component results are
clamped before integer division. All retained results are non-negative, so integer
division defines truncation toward zero without unspecified floating-point conversion.
A result exactly halfway between integers truncates to the lower integer.

The red-difference and blue-difference values for both source rows are converted before
averaging. Each vertical sum is divided by two with integer division; an exact half value
therefore truncates to the lower integer. No horizontal averaging occurs. The immutable
storage contains a 640 by 496 luma plane and two 640 by 248 colour-difference planes. This
is vertical 2:1 sampling, commonly described as 4:4:0, rather than Robot 36's frozen 4:2:0
storage.

The conversion acts on gamma-encoded sRGB byte values supplied by the safe image boundary;
it does not linearise them.

## Resolved disagreements

The original bulletin's nominal 2.08 ms porch and exact 126.103040-second image duration
are selected. QSSTV's 2.30 ms porch and adjusted duration are documented interoperability
settings and are not blended into the descriptor.

The Dayton limited-range nonlinear conversion and SSTV Encoder's truncation and
conversion-before-average behaviour form one attributable, independently implemented
definition. QSSTV's full-range approximation and average-before-conversion behaviour and
PySSTV's Pillow conversion are not used for golden colour values. The handbook's 480-line
active-image shorthand does not replace the evidenced 496 transmitted rows.

The compact vector under `tests/vectors/analogue/pd-120` independently calculates the
selected conversions, scans, boundaries, event count, duration, and frame counts without
importing or calling Scanline SSTV production code. `generate_reference.py` has SHA-256
`e66e4032d441c6d4c96e680a22a2fa75939cfceab830e1897eea6a602f72aaff` and
`reference.json` has SHA-256
`bda82ec62d236ab8f52274dccab4b719fee68b9c7ec467e7896fb66f1571ce5b`.

## Project waveform policy

The offline renderer uses the existing explicit amplitude of `0.8` for every event. This
is a project output-level choice, not an on-air protocol constant. Samples are not
normalised, limited, dithered, or automatically played. The accepted PCM16 conversion
and atomic WAV publication behaviour remain unchanged.
